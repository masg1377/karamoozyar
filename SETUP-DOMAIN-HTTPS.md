# 🌐 راه‌اندازی دامنه yar.mkrm.ir با HTTPS — راهنمای کامل قدم‌به‌قدم

این راهنما فرض می‌کند **هیچی بلد نیستی**. هر قدم را دقیق انجام بده و قبل از رفتن به قدم بعد،
بخش «چک کن» همان قدم را تست کن. اگر ارور گرفتی، بخش «اگر کار نکرد» همان قدم را بخوان.

> سرور: **Windows VPS** — آی‌پی `185.255.88.13`
> اپ تو الان به‌صورت native (با PM2) بالاست:
> - وب (Next.js) روی پورت **9100**
> - API (NestJS) روی پورت **9101** (شامل REST، فایل‌ها `/files`، و ریل‌تایم `/socket.io`)
> - Postgres/Redis/MinIO داخل Docker

# ⭐⭐ تشخیص نهایی (بعد از بررسی کامل سرور) — برو سراغ «حالت E»

بعد از دیدن سرور مشخص شد:
- دامنه پشت **آروان‌کلاد** است و آروان روی **پورت ۴۴۳ (HTTPS)** به سرورت وصل می‌شود (حالت SSL = Full).
- پورت ۸۰ سرور **آزاد** است (سایت Default Web Site متوقف است).
- روی سرور **IIS** نصب است با این سایت‌ها: `backend`(۵۰۰۰)، `ReactApp`=pwa(۴۴۳)، `admin`(۵۰۰۱).
- یکی از این سایت‌های IIS (همان که آروان `yar` را به آن می‌فرستد) به‌جای نشان‌دادن اپ،
  یک **ریدایرکت ۳۰۲** به `http://185.255.88.13:9100` می‌دهد. اپ خودت روی ۹۱۰۰/۹۱۰۱ سالم است.

چون آروان روی ۴۴۳ (که دست IIS است) وصل می‌شود، **راه‌حل تو «حالت E» است: IIS را با ARR
به reverse-proxy تبدیل کن** — نه Caddy. (حالت D فقط وقتی بود که آروان روی پورت ۸۰ وصل می‌شد.)

➡️ مستقیم برو به بخش **«🅴 حالت E»** پایین‌تر.

# (آرشیو) وضعیت پشت آروان‌کلاد با اتصال روی پورت ۸۰ (حالت D)

با تستی که گرفتیم مشخص شد دامنه‌ی `yar.mkrm.ir` **مستقیم روی سرور تو نیست**؛
**پشت آروان‌کلاد (ArvanCloud)** است:

- `yar.mkrm.ir` به IP های آروان (`185.143.233.235` / `185.143.234.235`) می‌خورد، نه به سرور تو.
- جوابِ `Server: ArvanCloud` و ریدایرکتِ `http → https` یعنی **خودِ آروان HTTPS را رایگان می‌دهد**.

**یعنی چه؟** معماری این است: `کاربر → HTTPS → آروان‌کلاد → سرور تو (origin)`.
پس:

1. ✅ **برای HTTPS کاری لازم نیست** — آروان خودش انجامش می‌دهد. (نقشه‌ی Caddy + Let's Encrypt
   که در قسمت‌های بعدی این فایل آمده، برای حالتی است که دامنه مستقیم روی سرور باشد؛ **مالِ تو نیست**.)
2. ✅ آن پرشِ زشت به `http://185.255.88.13:9100` از این می‌آید که **پورت ۸۰ سرورِ تو (origin)**
   به‌جای نشان‌دادن اپ، یک «ریدایرکت» به آن آدرس می‌دهد و آروان همان را به کاربر پاس می‌دهد.
   **این کاملاً از روی سرور خودت حل می‌شود** و به پنل آروان نیاز ندارد.

> راه‌حل در یک جمله: روی پورت ۸۰ سرور، به‌جای آن «ریدایرکت»، یک **reverse proxy تمیز (Caddy)**
> بگذار که اپ را مستقیم سرو کند. آن‌وقت آروان محتوای اپ را زیر `https://yar.mkrm.ir` نشان می‌دهد
> و دیگر پورت ۹۱۰۰ و آی‌پی دیده نمی‌شود. **اپ روی ۹۱۰۰ بماند هیچ اشکالی ندارد.**

برای این فایل یک `Caddyfile` مخصوصِ همین حالت (پشت آروان، روی پورت ۸۰، بدون SSL) آماده کرده‌ام.

## 🅳 قدم‌های حالت D (فقط روی سرور — بدون دسترسی به آروان/DNS)

### D-۱) اول دقیق بفهم پورت ۸۰ سرور الان دست کیست و چه جوابی می‌دهد
روی **سرور** در PowerShell ادمین:
```powershell
# چه برنامه‌ای روی 80 و 443 گوش می‌دهد؟
netstat -ano | findstr ":80 "
netstat -ano | findstr ":443 "

# پورت 80 لوکال چه جوابی می‌دهد؟ (همان چیزی که آروان می‌بیند)
curl.exe -v -I http://localhost:80

# مطمئن شو خودِ اپ سالم بالاست:
curl.exe -I http://localhost:9100
curl.exe -I http://localhost:9101/api/v1
pm2 status
```
- اگر آخرین ستونِ `netstat` یک PID داد، با `tasklist /FI "PID eq شماره"` ببین مالِ چیست
  (`w3wp.exe` = IIS).
- اگر `curl http://localhost:80` جوابِ `301` به `185.255.88.13:9100` داد → **مقصرِ اصلی همین است**؛
  در D-۳ جایش Caddy می‌گذاریم.
- اگر `localhost:9100` و `localhost:9101` جواب دادند یعنی اپ سالم است (خوب).

> 📌 این خروجی‌ها را برام بفرست تا اگر چیز غیرمنتظره‌ای بود قبل از تخریب پورت ۸۰ بگیریمش.

### D-۲) پورت ۸۰ را آزاد کن (آن ریدایرکت‌کننده را خاموش کن)
- اگر روی ۸۰ **IIS** بود و `pwa.mkrm.ir` هم روی همین IIS نیست:
  ```powershell
  iisreset /stop
  Set-Service W3SVC -StartupType Disabled
  ```
- اگر **pwa هم روی همین IIS است** → IIS را خاموش نکن؛ بگو تا راهِ کنار هم گذاشتنِ هر دو را بدهم
  (نباید pwa بشکند).
- اگر برنامه‌ی دیگری بود (node/nginx/…)، همان را stop/disable کن.

**چک کن:** دوباره `netstat -ano | findstr ":80 "` → باید **خالی** شود.

### D-۳) Caddy را نصب کن و Caddyfileِ مخصوص آروان را بگذار
۱. نصب Caddy: **قدم ۳** همین فایل را انجام بده (دانلود `caddy.exe` در `C:\Caddy`).
۲. محتوای فایلِ `Caddyfile` (که کنار پروژه ساختم) را در `C:\Caddy\Caddyfile` بگذار. خلاصه‌اش:
```caddyfile
{
	auto_https off
}
:80 {
	encode zstd gzip
	handle /api/*      { reverse_proxy 127.0.0.1:9101 }
	handle /files/*    { reverse_proxy 127.0.0.1:9101 }
	handle /socket.io/* { reverse_proxy 127.0.0.1:9101 }
	handle             { reverse_proxy 127.0.0.1:9100 }
}
```
> ⚠️ خط `auto_https off` مهم است: چون دامنه روی سرور تو نیست، نباید Caddy دنبال گرفتنِ
> گواهی برود (شکست می‌خورد). HTTPS کارِ آروان است.

۳. تست نوشتار و اجرای دستی:
```powershell
cd C:\Caddy
.\caddy.exe validate --config C:\Caddy\Caddyfile
.\caddy.exe run --config C:\Caddy\Caddyfile
```
پنجره را باز بگذار.

### D-۴) کد را با env جدید build کن و ری‌استارت کن
روی سرور، در پوشه‌ی پروژه (`pnpm build` لازم است چون `NEXT_PUBLIC_*` موقع build جا می‌افتد):
```powershell
cd C:\KarAmoozYar
git pull          # اگر env با git نمی‌آید، پنج مقدار جدول پایین را دستی ست کن
pnpm install
pnpm build
pm2 restart all
pm2 save
```

### D-۵) تست نهایی (از کامپیوتر خودت)
```powershell
curl.exe -v -L https://yar.mkrm.ir
```
باید آخرش **`200 OK`** بدهد و آدرس روی `https://yar.mkrm.ir` بماند (دیگر نه آی‌پی، نه `:9100`).
بعد در مرورگر `https://yar.mkrm.ir` را با **Ctrl+F5** باز کن.

### D-۶) دائمی‌کردن Caddy
**قدم ۷** همین فایل (سرویس با NSSM) را انجام بده تا بعد از ری‌استارت سرور هم خودکار بالا بیاید.

### ⚠️ دو نکته‌ی مخصوص آروان (اگر بعد از این چیزی کار نکرد)
- **ریل‌تایم/چت (WebSocket):** اگر آروان WebSocket را رد نکند، کتابخانه‌ی socket.io خودکار به
  حالت polling برمی‌گردد (در کد فعال است) و چت باز هم کار می‌کند — فقط ممکن است کمی کندتر باشد.
  اگر چت اصلاً کار نکرد، فعال‌کردن WebSocket در پنل آروان لازم می‌شود (کارِ کسی که دسترسی دارد).
- **خطای 502/525 بعد از راه‌اندازی:** یعنی حالت SSL آروان روی «Full» است و انتظار HTTPS از origin
  دارد. آن تنظیم در پنل آروان است؛ اگر به این خوردی بگو تا نسخه‌ی HTTPS-origin (با `tls internal`)
  را برایت بسازم.

> بقیه‌ی این فایل (قدم ۱ تا حالت C) برای حالتی است که دامنه **مستقیم** روی سرور باشد.
> چون تو پشت آروان هستی، فقط همین «حالت D» را دنبال کن؛ بخش‌های نصب Caddy (قدم ۳) و
> سرویس‌کردن (قدم ۷) مشترک‌اند.

# 🅴 حالت E — راه‌حل قطعی تو (IIS + ARR، بدون آسیب به pwa)

ایده: روی همان سایت IIS که آروان `yar` را به آن می‌فرستد، یک قانونِ **پراکسی** فقط برای
هاستِ `yar.mkrm.ir` می‌گذاریم تا به‌جای ریدایرکت، محتوای اپ (۹۱۰۰/۹۱۰۱) را برگرداند.
همه‌چیز روی سرور انجام می‌شود؛ به آروان و DNS دست نمی‌زنیم.

## STEP 1 — بفهم آروان `yar` را به کدام پورت می‌فرستد (و ARR هست یا نه)

روی سرور، PowerShell ادمین، **همهٔ این‌ها را یک‌جا** بزن:

```powershell
# کدام پورت IIS برای yar همان «302 → :9100» را می‌دهد؟
foreach ($p in 443,5000,5001) {
  "===== PORT $p ====="
  curl.exe -s -k -i -H "Host: yar.mkrm.ir" "https://127.0.0.1:$p/" |
    Select-String -Pattern "HTTP/|Location:|X-Powered|Server:"
}

# ARR نصب است یا نه؟
Get-WebGlobalModule | Where-Object { $_.Name -match "ApplicationRequestRouting" } | Select-Object Name
```

از خروجی: هر پورتی که `302` و `Location: http://185.255.88.13:9100` داد، **همان پورتِ yar است**.
این جدول می‌گوید بعداً کدام فایل را ویرایش کنی:

| پورتِ yar | سایت IIS | فایلی که ویرایش می‌کنی |
|---|---|---|
| `5000` | backend | `C:\publish\web.config` |
| `5001` | admin | `C:\admin\web.config` |
| `443` | ReactApp | `C:\pwa\web.config` |

## STEP 2 — نصب ARR (اگر در STEP 1 چیزی برنگشت یعنی نصب نیست)

۱. روی **مرورگرِ سرور** برو به: `https://www.iis.net/downloads/microsoft/application-request-routing`
   و فایلِ **ARR 3.0** را دانلود و نصب کن (نصبِ بعدی/بعدی). این ماژول، URL Rewrite را هم
   اگر نباشد می‌آورد (تو از قبل داری).
۲. بعد از نصب، **پراکسیِ ARR را در سطح سرور روشن کن** (یک‌بار کافی است):
```powershell
Set-WebConfigurationProperty -pspath 'MACHINE/WEBROOT/APPHOST' -filter "system.webServer/proxy" -name "enabled" -value "True"
```
**چک کن:** اگر این دستور بدون ارور اجرا شد، ARR درست نصب شده. اگر ارورِ «filter ... unrecognized»
داد، یعنی ARR هنوز نصب نشده — برگرد مرحله ۱.

## STEP 3 — اضافه‌کردن قوانینِ پراکسیِ yar (فقط همان فایلِ جدول بالا)

۱. **اول یک بکاپ بگیر** (مهم، تا اگر چیزی خراب شد برگردانی). فرض می‌کنیم پورت `5000` بود؛
   اگر فرق داشت مسیر را عوض کن:
```powershell
Copy-Item C:\publish\web.config C:\publish\web.config.bak
```
۲. همان فایل را با **Notepad** باز کن، تگِ `<rules>` را پیدا کن و **بلافاصله بعد از آن**،
   محتوای فایلِ `iis-yar-reverse-proxy-rules.xml` (که کنار پروژه ساختم) را پیست کن.
   اگر در فایل اصلاً تگِ `<rewrite><rules>` وجود نداشت، این ساختار را داخل `<system.webServer>` بگذار:
```xml
<rewrite>
  <rules>
    <!-- اینجا همان ۴ قانونِ YAR-* را پیست کن -->
  </rules>
</rewrite>
```
   خلاصهٔ ۴ قانون (اگر فایل را دمِ دستت نداری):
```xml
<rule name="YAR-API" stopProcessing="true">
  <match url="^(api/.*)" />
  <conditions><add input="{HTTP_HOST}" pattern="^yar\.mkrm\.ir$" /></conditions>
  <action type="Rewrite" url="http://127.0.0.1:9101/{R:1}" />
</rule>
<rule name="YAR-FILES" stopProcessing="true">
  <match url="^(files/.*)" />
  <conditions><add input="{HTTP_HOST}" pattern="^yar\.mkrm\.ir$" /></conditions>
  <action type="Rewrite" url="http://127.0.0.1:9101/{R:1}" />
</rule>
<rule name="YAR-SOCKETIO" stopProcessing="true">
  <match url="^(socket\.io/.*)" />
  <conditions><add input="{HTTP_HOST}" pattern="^yar\.mkrm\.ir$" /></conditions>
  <action type="Rewrite" url="http://127.0.0.1:9101/{R:1}" />
</rule>
<rule name="YAR-APP" stopProcessing="true">
  <match url="^(.*)" />
  <conditions><add input="{HTTP_HOST}" pattern="^yar\.mkrm\.ir$" /></conditions>
  <action type="Rewrite" url="http://127.0.0.1:9100/{R:1}" />
</rule>
```
۳. ذخیره کن، بعد IIS را ری‌استارت کن:
```powershell
iisreset
```

حالا باید **آدرس روی `https://yar.mkrm.ir` بماند** و آخرش `200 OK` با HTMLِ کارآموزیار بیاید
(دیگر نه ۳۰۲، نه آی‌پی، نه `:9100`). بعد در مرورگر `https://yar.mkrm.ir` را با **Ctrl+F5** باز کن،
لاگین و چت و عکس را تست کن.

## اگر کار نکرد
- باز هم `302 → :9100` می‌گیری → یعنی قوانین در فایلِ اشتباهی پیست شده، یا پورت در STEP 1 را
  اشتباه خوانده‌ای. فایلِ همان پورتی که `302` داد را ویرایش کن.
- خطای **500.x** بعد از پیست → نوشتارِ XMLِ web.config خراب شده؛ فایل بکاپ را برگردان:
  `Copy-Item C:\publish\web.config.bak C:\publish\web.config -Force; iisreset` و دوباره دقیق پیست کن.
- صفحه می‌آید ولی **چت ریل‌تایم کار نمی‌کند** → احتمالاً IIS فیچرِ WebSocket را ندارد؛
  socket.io خودش به polling برمی‌گردد و باز کار می‌کند. برای WebSocketِ کامل، فیچرِ
  «WebSocket Protocol» ویندوز را نصب کن (Server Manager → Add Roles → Web Server → WebSocket Protocol).
- **pwa.mkrm.ir** را هم یک‌بار تست کن که سالم مانده باشد (باید باشد، چون قوانین فقط شرطِ
  هاستِ yar دارند).

## 🔎 چرا الان به `http://185.255.88.13:9100/` می‌پره؟

وقتی `yar.mkrm.ir` می‌زنی و **آدرسِ مرورگر عوض می‌شود** و می‌رود روی آی‌پی و پورت،
یعنی یک «ریدایرکت/فوروارد» در کار است. تقریباً همیشه یکی از این دوتاست:

1. **URL Forwarding در پنل DNS** (محتمل‌ترین): توی پنل دامنه، برای `yar.mkrm.ir`
   به‌جای یک رکورد A، یک «Redirect/URL Forward» ساخته شده که می‌گوید برو به
   `http://185.255.88.13:9100`. این دقیقاً همان رفتاری است که می‌بینی (آدرس بار عوض می‌شود).
2. یک وب‌سروری روی خود سرور (مثلاً IIS) که این ریدایرکت را انجام می‌دهد.

راه‌حل درست: دامنه را با یک **رکورد A** مستقیم به آی‌پی وصل کنیم، روی سرور یک
**reverse proxy** (برنامه‌ی Caddy) بگذاریم که روی پورت ۴۴۳ (HTTPS) گوش بدهد و درخواست‌ها را
بی‌سروصدا به اپ‌های داخلی (9100 و 9101) پاس بدهد. اینطوری آدرس روی `https://yar.mkrm.ir`
**می‌ماند**، قفل سبز HTTPS هم فعال می‌شود.

چرا Caddy؟ چون روی ویندوز ساده است و **خودش گواهی SSL را رایگان از Let's Encrypt می‌گیرد و
هر ۳ ماه خودکار تمدید می‌کند** — تو کاری نمی‌کنی.

## 🔑 «به DNS دسترسی ندارم — میشه فقط از سرور درستش کنم؟»

جوابِ کوتاه: **بستگی دارد به اینکه ترافیکِ `yar.mkrm.ir` اصلاً به سرور تو می‌رسد یا نه.**
یک رکورد A نمی‌تواند پورت داشته باشد؛ پس دیدنِ `:9100` یعنی یک «ریدایرکت» در کار است.
این یک تستِ ۱۰ ثانیه‌ای تکلیف را روشن می‌کند.

### تستِ تعیین‌کننده (روی **کامپیوتر خودت**، نه سرور — CMD یا PowerShell):

```powershell
nslookup yar.mkrm.ir
```
به خطِ **Address** در جواب نگاه کن:

- **اگر `185.255.88.13` بود** → ✅ ترافیک به سرور تو می‌رسد. یعنی آن پرش به `:9100` کارِ یک
  برنامهٔ روی **پورت ۸۰ خودِ سرور** است. **بله، کاملاً از روی سرور حل می‌شود، بدون تغییر DNS.**
  → برو سراغ **حالت C** پایین.

- **اگر یک IP دیگر بود** (هر چیزی غیر از 185.255.88.13) → ❌ این یک «URL Forwarding» در پنل
  ثبت‌کنندهٔ دامنه است و ریدایرکت **قبل از رسیدن به سرور تو** انجام می‌شود. از سرور
  نمی‌شود کاری کرد. فقط کسی که به DNS دسترسی دارد باید این یک کار را بکند:
  > «رکوردِ URL Forwarding / Redirect برای `yar` را حذف کن و به‌جایش یک رکورد **A** بساز:
  >  Host = `yar` ، Value = `185.255.88.13`.»

  این پیام را عیناً برای ادمینِ DNS بفرست. بعد از این تغییر، بقیهٔ مراحل سرور کار می‌کنند.

> برای اطمینان بیشتر، این را هم بزن (ویندوز ۱۰ به بعد `curl` دارد):
> ```powershell
> curl.exe -v -I http://yar.mkrm.ir
> ```
> اگر در خروجی دیدی `Trying 185.255.88.13...` یعنی به سرور تو وصل شده ✅ (حالت C).
> اگر `Trying <یک IP دیگر>` بود، یعنی فوروارد ثبت‌کننده است ❌.

# قدم ۱ — رکورد DNS را درست کن (مهم‌ترین قدم)

> 🟢 **اگر تستِ بالا نشان داد `yar.mkrm.ir` به `185.255.88.13` resolve می‌شود، این قدم را
> رد کن و مستقیم برو «حالت C» (پایین صفحه).** DNS از قبل درست است و دسترسی به پنل لازم نداری.
> این قدم فقط برای وقتی است که خودت به پنل DNS دسترسی داری و باید فوروارد را به A تبدیل کنی.

برو به **پنل مدیریت DNS دامنه‌ی mkrm.ir** (همان‌جایی که رکوردها را می‌سازی).

۱. هر **«URL Forwarding»** یا **«Redirect»** یا **«Web Forwarding»** که برای `yar` وجود دارد را
   **حذف کن**. (این همان چیزی است که الان تو را به آی‌پی:9100 می‌فرستد.)
۲. یک **رکورد A** بساز:

| Type | Name / Host | Value / Points to | TTL |
|---|---|---|---|
| `A` | `yar` | `185.255.88.13` | `3600` (یا Auto) |

> نکته: در فیلد Name فقط `yar` را بنویس، نه `yar.mkrm.ir` (پنل خودش دامنه را اضافه می‌کند).
> اگر پنل اجازه‌ی `@`/CNAME خواست، کاری نداشته باش؛ فقط همین رکورد A لازم است.

**چک کن:** چند دقیقه (تا حداکثر ~۱ ساعت) صبر کن تا DNS پخش شود، بعد روی **کامپیوتر خودت**
(نه سرور) در PowerShell یا CMD بزن:

```powershell
nslookup yar.mkrm.ir
```
باید در جواب `185.255.88.13` را ببینی.

**اگر کار نکرد:**
- هنوز آی‌پی قدیمی یا «forward» را نشان می‌دهد → احتمالاً فوروارد را کامل حذف نکرده‌ای، یا
  هنوز TTL قبلی منقضی نشده. کمی بیشتر صبر کن و دوباره بزن.
- `nslookup` می‌گوید «can't find» → رکورد A هنوز ساخته/ذخیره نشده. پنل را دوباره چک کن.

# قدم ۴ — فایل پیکربندی Caddy را بگذار

من یک فایل آماده به اسم `Caddyfile` کنار پروژه ساختم (در ریشه‌ی KarAmoozYar).
محتوای آن را در سرور در مسیر `C:\Caddy\Caddyfile` کپی کن.

ساده‌ترین راه: روی سرور این دستور را بزن تا فایل ساخته شود (یا با Notepad دستی بساز):

```powershell
@'
yar.mkrm.ir {
	encode zstd gzip

	handle /api/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle /files/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle /socket.io/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle {
		reverse_proxy 127.0.0.1:9100
	}
}
'@ | Out-File -FilePath C:\Caddy\Caddyfile -Encoding utf8
```

> این یعنی: درخواست‌های `/api`، `/files` و `/socket.io` برو به API (9101)، بقیه برو به وب (9100).
> Caddy خودش WebSocketِ ریل‌تایم را هم درست رد می‌کند — تنظیم اضافه لازم نیست.

**چک کن — درستیِ نوشتارِ فایل را تست بگیر:**
```powershell
cd C:\Caddy
.\caddy.exe validate --config C:\Caddy\Caddyfile
```
باید بنویسد `Valid configuration`.

**اگر کار نکرد:** اگر ارور نوشتاری داد، یعنی موقع کپی، فاصله/تب یا براکت `{ }` خراب شده.
دوباره از روی بلوک بالا کپی کن. (در Caddyfile دندانه‌ها باید با **Tab** باشند.)

# 🅱️ حالت B — اگر IIS الان پورت ۸۰/۴۴۳ را گرفته و pwa.mkrm.ir را بالا نگه داشته

در این حالت نمی‌توانی هم‌زمان IIS و Caddy را روی پورت ۴۴۳ بگذاری (تداخل پورت). دو انتخاب داری:

### گزینه‌ی B-1 (پیشنهادی): همه‌چیز را به Caddy بسپار
Caddy می‌تواند **هر دو دامنه** را اداره کند. فقط باید بدانی اپ `pwa` روی چه پورتی داخل سرور
بالاست.

۱. ببین IIS سایت pwa را به کجا می‌فرستد / اپ pwa روی چه پورتی است (مثلاً 3000، 8080، ...).
۲. IIS را خاموش کن تا پورت‌ها آزاد شوند:
```powershell
iisreset /stop
Set-Service W3SVC -StartupType Disabled
```
۳. در `C:\Caddy\Caddyfile` بلوک pwa را هم اضافه کن (PORT_PWA را با پورت واقعی عوض کن):
```
pwa.mkrm.ir {
	encode zstd gzip
	reverse_proxy 127.0.0.1:PORT_PWA
}
```
۴. برگرد قدم‌های ۴ تا ۷ را انجام بده. حالا Caddy برای **هر دو** دامنه گواهی می‌گیرد.

> اگر پورت داخلیِ pwa را نمی‌دانی، بگو تا کمکت کنم پیدایش کنیم (یا همان `netstat`/IIS Manager).

### گزینه‌ی B-2: IIS را نگه دار و yar را هم با IIS بالا بیاور
اگر نمی‌خواهی IIS را دست بزنی، می‌شود با ماژول‌های **URL Rewrite + ARR** یک reverse proxy در IIS
برای `yar.mkrm.ir` ساخت و گواهی SSL را با **win-acme** گرفت. این روش کلیک‌های بیشتری دارد و
برای کسی که تازه‌کار است سخت‌تر است؛ اگر این مسیر را خواستی بگو تا قدم‌به‌قدم برایت بنویسم.

# 🅲 حالت C — به DNS دسترسی نداری ولی `yar.mkrm.ir` به سرور تو می‌رسد (رایج‌ترین حالت تو)

این حالت برای وقتی است که تستِ `nslookup yar.mkrm.ir` جواب **`185.255.88.13`** داد.
یعنی DNS از قبل درست به سرور تو اشاره می‌کند و **به پنل DNS اصلاً نیازی نداری**. آن پرشِ
زشت به `:9100` کارِ یک برنامه روی **پورت ۸۰ خودِ سرور** است؛ کافی است آن را با Caddy عوض کنیم.

> مهم: اپ تو روی پورت ۹۱۰۰ بماند هیچ اشکالی ندارد و درست هم هست. ما اپ را جابه‌جا نمی‌کنیم؛
> فقط یک «درِ ورودیِ تمیز» (Caddy) جلوش روی پورت‌های ۸۰/۴۴۳ می‌گذاریم که آدرس را روی
> `https://yar.mkrm.ir` نگه دارد و کاربر دیگر پورت ۹۱۰۰ را نبیند.

## C-۱) ببین چه برنامه‌ای روی پورت ۸۰ دارد ریدایرکت می‌کند

روی **سرور** در PowerShell ادمین:

```powershell
netstat -ano | findstr ":80 "
netstat -ano | findstr ":443 "
```
اگر چیزی برگشت، آخرین ستون PID است؛ ببین مالِ چه برنامه‌ای است:
```powershell
tasklist /FI "PID eq شماره_PID"
```
- اگر `w3wp.exe` یا سرویس `World Wide Web Publishing` بود → **IIS** است.
- ممکن است یک nginx/caddy/node قدیمی یا یک «redirect site» باشد.

## C-۲) آن ریدایرکت‌کننده را خاموش/آزاد کن (تا پورت ۸۰ و ۴۴۳ آزاد شود)

- اگر **IIS** بود و فقط داشت yar را ریدایرکت می‌کرد (یعنی pwa روی IIS نیست):
```powershell
iisreset /stop
Set-Service W3SVC -StartupType Disabled
```
- اگر **IIS** بود ولی **pwa.mkrm.ir هم روی همان IIS است** → IIS را خاموش نکن؛
  برو **حالت B** (آنجا گفته شده چطور هر دو دامنه را با هم به Caddy بسپاری بدون اینکه pwa بشکند).
- اگر برنامهٔ دیگری بود (مثلاً یک node یا nginx کوچک) → همان را stop/disable کن.

**چک کن:** دوباره `netstat -ano | findstr ":80 "` بزن؛ حالا باید **خالی** باشد.

## C-۳) Caddy را نصب و راه‌اندازی کن

دقیقاً همان **قدم‌های ۲ تا ۷** این راهنما را انجام بده:
- قدم ۲: فایروال ویندوز (و فایروال پنل VPS) پورت ۸۰ و ۴۴۳.
- قدم ۳: نصب `caddy.exe` در `C:\Caddy`.
- قدم ۴: گذاشتن `Caddyfile` (همان فایل آماده).
- قدم ۵: یک‌بار `caddy run` تا گواهی HTTPS برای `yar.mkrm.ir` گرفته شود و تست.
- قدم ۶: روی سرور `git pull` → `pnpm build` → `pm2 restart all` (به‌خاطر env های جدید).
- قدم ۷: Caddy را با NSSM سرویس دائمی کن.

> چون DNS از قبل به سرور می‌خورد، Let's Encrypt راحت گواهی می‌دهد و **هیچ کاری با DNS لازم نیست**.

**اگر در قدم ۵ گواهی گرفته نشد** و در لاگ Caddy خطای `timeout`/`connection refused` دیدی:
معمولاً یعنی پورت ۸۰ از بیرون بسته است → فایروال **پنل VPS** را هم چک کن (قدم ۲).
**اگر `bind: address already in use` داد** → یعنی هنوز برنامهٔ قبلی پورت ۸۰/۴۴۳ را ول نکرده →
برگرد C-۲.

> ❗ تنها حالتی که حالت C جواب نمی‌دهد: اگر `nslookup` یک IP **غیر از** 185.255.88.13 برگرداند
> (فوروارد در سطح ثبت‌کنندهٔ دامنه). آن موقع چاره‌ای جز تغییر DNS نیست؛ پیامِ آمادهٔ بخش
> «🔑 به DNS دسترسی ندارم» را برای ادمینِ DNS بفرست.
