# 🌐 راه‌اندازی دامنه yar.mkrm.ir با HTTPS — راهنمای کامل قدم‌به‌قدم

این راهنما فرض می‌کند **هیچی بلد نیستی**. هر قدم را دقیق انجام بده و قبل از رفتن به قدم بعد،
بخش «چک کن» همان قدم را تست کن. اگر ارور گرفتی، بخش «اگر کار نکرد» همان قدم را بخوان.

> سرور: **Windows VPS** — آی‌پی `185.255.88.13`
> اپ تو الان به‌صورت native (با PM2) بالاست:
> - وب (Next.js) روی پورت **9100**
> - API (NestJS) روی پورت **9101** (شامل REST، فایل‌ها `/files`، و ریل‌تایم `/socket.io`)
> - Postgres/Redis/MinIO داخل Docker

---

## 🔎 چرا الان به `http://185.255.88.13:9100/` می‌پره؟

وقتی `yar.mkrm.ir` می‌زنی و **آدرسِ مرورگر عوض می‌شود** و می‌رود روی آی‌پی و پورت،
یعنی یک «ریدایرکت/فوروارد» در کار است. تقریباً همیشه یکی از این دوتاست:

1. **URL Forwarding در پنل DNS** (محتمل‌ترین): توی پنل دامنه، برای `yar.mkrm.ir`
   به‌جای یک رکورد A، یک «Redirect/URL Forward» ساخته شده که می‌گوید برو به
   `http://185.255.88.13:9100`. این دقیقاً همان رفتاری است که می‌بینی (آدرس بار عوض می‌شود).
2. یک وب‌سروری روی خود سرور (مثلاً IIS) که این ریدایرکت را انجام می‌دهد.

راه‌حل درست: دامنه را با یک **رکورد A** مستقیم به آی‌پی وصل کنیم، روی سرور یک
**reverse proxy** (برنامه‌ی Caddy) بگذاریم که روی پورت ۴۴۳ (HTTPS) گوش بدهد و درخواست‌ها را
بی‌سروصدا به اپ‌های داخلی (9100 و 9101) پاس بدهد. اینطوری آدرس روی `https://yar.mkrm.ir`
**می‌ماند**، قفل سبز HTTPS هم فعال می‌شود.

چرا Caddy؟ چون روی ویندوز ساده است و **خودش گواهی SSL را رایگان از Let's Encrypt می‌گیرد و
هر ۳ ماه خودکار تمدید می‌کند** — تو کاری نمی‌کنی.

این‌ها را من توی کد عوض کردم؛ فقط باید روی سرور `git pull` و دوباره build/restart کنی (قدم ۶):

# 🧭 قدم ۰ — به سرور وصل شو و وضعیت فعلی را بفهم

۱. با **Remote Desktop (RDP)** به ویندوز سرور وصل شو (همان روش همیشگی‌ات).
۲. **PowerShell** را **به‌صورت Administrator** باز کن:
   دکمه Start → تایپ کن `PowerShell` → راست‌کلیک → **Run as administrator**.

حالا ببین چه چیزی روی پورت‌های ۸۰ و ۴۴۳ گوش می‌دهد (این تعیین می‌کند مسیر A را بروی یا B):

```powershell
netstat -ano | findstr ":80 "
netstat -ano | findstr ":443 "
```

**چک کن — نتیجه را اینطور بخوان:**
- اگر **هیچی برنگرداند** → پورت‌ها آزادند → **حالت A** (ساده). برو قدم ۱.
- اگر چیزی برگرداند، آخرین ستون یک عدد (PID) است. ببین مال چه برنامه‌ای است:

```powershell
tasklist /FI "PID eq شماره_PID"
```
  - اگر اسمش چیزی مثل `w3wp.exe` یا سرویس `World Wide Web Publishing` بود → **IIS** فعال است
    و احتمالاً همان است که `pwa.mkrm.ir` را بالا نگه داشته → **حالت B** را بخوان (پایین‌تر).

> اگر مطمئن نیستی pwa الان با چی بالا می‌آید، همین حالا مرورگر را باز کن و `pwa.mkrm.ir` را بزن
> تا ببینی اصلاً کار می‌کند یا نه. اگر pwa هم فقط یک فوروارد است (نه یک اپ واقعی روی این سرور)،
> خیالت راحت — حالت A را برو.

# قدم ۱ — رکورد DNS را درست کن (مهم‌ترین قدم)

برو به **پنل مدیریت DNS دامنه‌ی mkrm.ir** (همان‌جایی که رکوردها را می‌سازی).

۱. هر **«URL Forwarding»** یا **«Redirect»** یا **«Web Forwarding»** که برای `yar` وجود دارد را
   **حذف کن**. (این همان چیزی است که الان تو را به آی‌پی:9100 می‌فرستد.)
۲. یک **رکورد A** بساز:

| Type | Name / Host | Value / Points to | TTL |
|---|---|---|---|
| `A` | `yar` | `185.255.88.13` | `3600` (یا Auto) |

> نکته: در فیلد Name فقط `yar` را بنویس، نه `yar.mkrm.ir` (پنل خودش دامنه را اضافه می‌کند).
> اگر پنل اجازه‌ی `@`/CNAME خواست، کاری نداشته باش؛ فقط همین رکورد A لازم است.

**چک کن:** چند دقیقه (تا حداکثر ~۱ ساعت) صبر کن تا DNS پخش شود، بعد روی **کامپیوتر خودت**
(نه سرور) در PowerShell یا CMD بزن:

```powershell
nslookup yar.mkrm.ir
```
باید در جواب `185.255.88.13` را ببینی.

**اگر کار نکرد:**
- هنوز آی‌پی قدیمی یا «forward» را نشان می‌دهد → احتمالاً فوروارد را کامل حذف نکرده‌ای، یا
  هنوز TTL قبلی منقضی نشده. کمی بیشتر صبر کن و دوباره بزن.
- `nslookup` می‌گوید «can't find» → رکورد A هنوز ساخته/ذخیره نشده. پنل را دوباره چک کن.

# قدم ۲ — فایروال ویندوز را برای پورت‌های ۸۰ و ۴۴۳ باز کن

روی **سرور** در PowerShell ادمین:

```powershell
New-NetFirewallRule -DisplayName "HTTP 80"  -Direction Inbound -Protocol TCP -LocalPort 80  -Action Allow
New-NetFirewallRule -DisplayName "HTTPS 443" -Direction Inbound -Protocol TCP -LocalPort 443 -Action Allow
```

**چک کن:**
```powershell
Get-NetFirewallRule -DisplayName "HTTP 80","HTTPS 443" | Format-Table DisplayName,Enabled,Direction,Action
```
باید هر دو `Enabled=True` و `Action=Allow` باشند.

> ⚠️ بعضی ارائه‌دهنده‌های VPS یک **فایروال بیرونی** (در پنل خودشان) هم دارند. اگر داری،
> آنجا هم پورت‌های 80 و 443 را روی Allow بگذار، وگرنه Let's Encrypt نمی‌تواند گواهی بدهد.

# قدم ۳ — Caddy را نصب کن

ساده‌ترین راه دانلود مستقیم فایل اجرایی است.

۱. روی سرور پوشه بساز:

۲. فایل `caddy.exe` را دانلود کن. **مرورگرِ روی سرور** را باز کن و برو به:
   `https://caddyserver.com/download`
   - گزینه‌ها: Platform = **windows**، Arch = **amd64** → دکمه **Download** را بزن.
   - فایل دانلودشده را به اسم `caddy.exe` تغییر بده و داخل `C:\Caddy` بگذار.

   (اگر [Chocolatey](https://chocolatey.org) روی سرور نصب است، فقط همین یک خط هم کافی است:

**چک کن:**
```powershell
C:\Caddy\caddy.exe version
```
باید یک شماره‌ی نسخه مثل `v2.x.x` چاپ شود.

**اگر کار نکرد:** اگر گفت فایل پیدا نشد، یعنی `caddy.exe` در `C:\Caddy` نیست یا اسمش فرق دارد؛
دوباره فایل را همان‌جا با همان اسم بگذار.

# قدم ۴ — فایل پیکربندی Caddy را بگذار

من یک فایل آماده به اسم `Caddyfile` کنار پروژه ساختم (در ریشه‌ی KarAmoozYar).
محتوای آن را در سرور در مسیر `C:\Caddy\Caddyfile` کپی کن.

ساده‌ترین راه: روی سرور این دستور را بزن تا فایل ساخته شود (یا با Notepad دستی بساز):

```powershell
@'
yar.mkrm.ir {
	encode zstd gzip

	handle /api/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle /files/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle /socket.io/* {
		reverse_proxy 127.0.0.1:9101
	}
	handle {
		reverse_proxy 127.0.0.1:9100
	}
}
'@ | Out-File -FilePath C:\Caddy\Caddyfile -Encoding utf8
```

> این یعنی: درخواست‌های `/api`، `/files` و `/socket.io` برو به API (9101)، بقیه برو به وب (9100).
> Caddy خودش WebSocketِ ریل‌تایم را هم درست رد می‌کند — تنظیم اضافه لازم نیست.

**چک کن — درستیِ نوشتارِ فایل را تست بگیر:**
```powershell
cd C:\Caddy
.\caddy.exe validate --config C:\Caddy\Caddyfile
```
باید بنویسد `Valid configuration`.

**اگر کار نکرد:** اگر ارور نوشتاری داد، یعنی موقع کپی، فاصله/تب یا براکت `{ }` خراب شده.
دوباره از روی بلوک بالا کپی کن. (در Caddyfile دندانه‌ها باید با **Tab** باشند.)

# قدم ۵ — یک‌بار دستی تست کن که گواهی HTTPS گرفته شود

قبل از اینکه سرویس دائمی بسازیم، یک‌بار دستی اجرا کن تا مطمئن شویم گواهی صادر می‌شود:

```powershell
cd C:\Caddy
.\caddy.exe run --config C:\Caddy\Caddyfile
```

پنجره را **باز بگذار**. Caddy تلاش می‌کند برای `yar.mkrm.ir` گواهی بگیرد.
حالا از مرورگرِ کامپیوتر خودت برو به: **https://yar.mkrm.ir**

**چک کن:**
- آدرس روی `https://yar.mkrm.ir` می‌ماند، قفل سبز دارد، و اپ بالا می‌آید → 🎉 عالی!
  حالا در همان پنجره‌ی PowerShell `Ctrl + C` بزن تا متوقف شود و برو قدم ۶ (سرویس دائمی).

**اگر کار نکرد:**
- در لاگ‌های Caddy دنبال کلمه‌ی `certificate` یا `error` بگرد:
  - خطای مربوط به Let's Encrypt / `timeout` / `connection refused` → یعنی Let's Encrypt نتوانست
    از بیرون به پورت ۸۰ سرور برسد. برگرد قدم ۲ (فایروال ویندوز **و** فایروال پنل VPS) و قدم ۱
    (رکورد A باید دقیقاً به `185.255.88.13` اشاره کند).
  - `bind: address already in use` روی پورت 80/443 → یعنی برنامه‌ی دیگری (مثل IIS) پورت را
    گرفته. برو **حالت B** پایین.
- صفحه باز می‌شود ولی **502 Bad Gateway** می‌دهد → یعنی Caddy سالم است ولی اپ داخلی بالا نیست.
  چک کن اپ‌ها روشن‌اند:
  ```powershell
  pm2 status
  ```
  باید `karamooz-web` (9100) و `karamooz-api` (9101) هر دو `online` باشند. اگر نه: `pm2 restart all`.

# قدم ۶ — کد جدید را بگیر، دوباره build و restart کن

چون مقادیر `NEXT_PUBLIC_*` موقع **build** داخل کلاینت جا می‌افتند، حتماً وب باید دوباره build شود.

روی سرور، در پوشه‌ی پروژه:

```powershell
cd C:\KarAmoozYar        # مسیر واقعی پروژه را بگذار

pnpm install              # برای اطمینان
pnpm build                # هم وب و هم API را build می‌کند (Turborepo)
pm2 restart all           # هر دو اپ را با env جدید بالا بیاور
pm2 save
```

> اگر env را با git مدیریت نمی‌کنی (چون `.env` ها معمولاً در `.gitignore` هستند)، باید همان
> پنج مقدار جدول بالا را **دستی** روی سرور در `apps/web/.env.local` و `apps/api/.env` ویرایش کنی،
> بعد `pnpm build` و `pm2 restart all`.

**چک کن:**
```powershell
pm2 status
```
هر دو `online`. بعد در مرورگر `https://yar.mkrm.ir` را باز کن، یک‌بار **Ctrl+F5** (رفرش کامل) بزن.

**اگر کار نکرد (چت/فایل/عکس نمی‌آید):**
- `F12` مرورگر → تب **Console** و **Network** را ببین.
  - اگر خطای `Mixed Content` یا درخواست به `http://185.255.88.13:9101` دیدی → یعنی وب با env
    قدیمی build شده. دوباره `pnpm build` و `pm2 restart karamooz-web`.
  - اگر `CORS` دیدی → مطمئن شو `apps/api/.env` مقدار `CORS_ORIGINS` شامل `https://yar.mkrm.ir`
    است، بعد `pm2 restart karamooz-api`.

# قدم ۷ — Caddy را به‌صورت سرویس دائمی دربیاور (تا بعد از ری‌استارت سرور هم بالا بماند)

اگر فقط `caddy run` بزنی، با بستن پنجره خاموش می‌شود. برای همیشگی، از **NSSM** استفاده می‌کنیم.

۱. NSSM را از `https://nssm.cc/download` دانلود کن، فایل `nssm.exe` (نسخه‌ی win64) را در `C:\Caddy`
   بگذار.
۲. سرویس را بساز:
```powershell
cd C:\Caddy
.\nssm.exe install Caddy "C:\Caddy\caddy.exe" "run --config C:\Caddy\Caddyfile"
.\nssm.exe set Caddy AppDirectory C:\Caddy
.\nssm.exe set Caddy Start SERVICE_AUTO_START
.\nssm.exe start Caddy
```

**چک کن:**
```powershell
.\nssm.exe status Caddy
```
باید `SERVICE_RUNNING` باشد. حالا `https://yar.mkrm.ir` باید کار کند، و حتی بعد از
**ری‌استارت سرور** هم خودکار بالا می‌آید.

**اگر کار نکرد:**
- `SERVICE_PAUSED` یا فوراً stop شد → یعنی پورت گرفته شده یا Caddyfile مشکل دارد. لاگ سرویس:
  ```powershell
  Get-Content C:\Caddy\caddy.log -Tail 50   # اگر لاگ به فایل می‌رود
  ```
  یا دوباره دستیِ قدم ۵ را اجرا کن تا ارور را زنده ببینی.
- اگر `nssm` نخواستی، جایگزین: در **Task Scheduler** یک تسک با تریگر «At startup» بساز که
  `C:\Caddy\caddy.exe run --config C:\Caddy\Caddyfile` را اجرا کند (Run whether user is logged on or not).

# 🅱️ حالت B — اگر IIS الان پورت ۸۰/۴۴۳ را گرفته و pwa.mkrm.ir را بالا نگه داشته

در این حالت نمی‌توانی هم‌زمان IIS و Caddy را روی پورت ۴۴۳ بگذاری (تداخل پورت). دو انتخاب داری:

### گزینه‌ی B-1 (پیشنهادی): همه‌چیز را به Caddy بسپار
Caddy می‌تواند **هر دو دامنه** را اداره کند. فقط باید بدانی اپ `pwa` روی چه پورتی داخل سرور
بالاست.

۱. ببین IIS سایت pwa را به کجا می‌فرستد / اپ pwa روی چه پورتی است (مثلاً 3000، 8080، ...).
۲. IIS را خاموش کن تا پورت‌ها آزاد شوند:
```powershell
iisreset /stop
Set-Service W3SVC -StartupType Disabled
```
۳. در `C:\Caddy\Caddyfile` بلوک pwa را هم اضافه کن (PORT_PWA را با پورت واقعی عوض کن):
```
pwa.mkrm.ir {
	encode zstd gzip
	reverse_proxy 127.0.0.1:PORT_PWA
}
```
۴. برگرد قدم‌های ۴ تا ۷ را انجام بده. حالا Caddy برای **هر دو** دامنه گواهی می‌گیرد.

> اگر پورت داخلیِ pwa را نمی‌دانی، بگو تا کمکت کنم پیدایش کنیم (یا همان `netstat`/IIS Manager).

### گزینه‌ی B-2: IIS را نگه دار و yar را هم با IIS بالا بیاور
اگر نمی‌خواهی IIS را دست بزنی، می‌شود با ماژول‌های **URL Rewrite + ARR** یک reverse proxy در IIS
برای `yar.mkrm.ir` ساخت و گواهی SSL را با **win-acme** گرفت. این روش کلیک‌های بیشتری دارد و
برای کسی که تازه‌کار است سخت‌تر است؛ اگر این مسیر را خواستی بگو تا قدم‌به‌قدم برایت بنویسم.

# 📌 جمع‌بندی خیلی کوتاه

1. در پنل DNS: فوروارد `yar` را حذف کن، رکورد **A → 185.255.88.13** بساز.
2. فایروال ویندوز (و پنل VPS): پورت **80 و 443** را باز کن.
3. **Caddy** را در `C:\Caddy` نصب کن، `Caddyfile` را بگذار.
4. یک‌بار `caddy run` بزن تا گواهی HTTPS گرفته شود و تست کن.
5. روی سرور: `git pull` → `pnpm build` → `pm2 restart all` (به‌خاطر env های جدید).
6. Caddy را با **NSSM** سرویس دائمی کن.
7. اگر IIS پورت‌ها را گرفته (به‌خاطر pwa) → «حالت B» را برو.

هر قدمی ارور داد، متنِ دقیق ارور را برام بفرست تا همان‌جا حلش کنیم.
